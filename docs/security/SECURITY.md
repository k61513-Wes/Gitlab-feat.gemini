# 安全規範 — GitLab Issue 整理工具

**對應版本：v1.2.0（文件拆分版）**
**最後更新：2026-04-17**

---

## 1. 目的

定義本專案在認證資訊、敏感資料、日誌與錯誤訊息上的最低安全要求。

---

## 2. 敏感資訊範圍

以下資料視為敏感資訊：

- GitLab API Token
- GitLab 帳號密碼
- Authorization Header
- 外部服務回傳中的憑證片段
- 可能含個資的 Issue 私密內容

---

## 3. 儲存與生命週期

### 3.1 前端
- Token 與密碼僅允許存在記憶體
- 禁止寫入 `localStorage`、`sessionStorage`、cookie
- 頁面刷新後需清空認證資訊
- `localStorage` 僅可保存非敏感 UI 偏好，例如主題、字體大小與既有 Project ID 偏好

### 3.2 後端
- 僅在請求生命週期中使用 token
- 禁止落盤到任何檔案（含 outputs）
- 禁止持久化至設定檔、快取或資料庫

---

## 4. 日誌與錯誤處理

### 4.1 禁止明文紀錄
禁止在 log、traceback、錯誤訊息中輸出以下欄位原文：
- `token`
- `password`
- `Authorization`

### 4.2 遮罩規則
- 若需除錯顯示，僅可顯示前後少量字元，其餘以 `***` 取代
- 禁止完整回傳第三方錯誤內文到前端

### 4.3 錯誤回應
- 依 `docs/specs/API_SPEC.md` 統一錯誤格式
- 對外訊息應可理解但不得洩漏內部細節

---

## 5. 外部程序呼叫安全

- 呼叫 Gemini CLI 時，禁止使用會提高注入風險的命令拼接策略
- 參數需以安全方式傳遞，避免將使用者輸入直接拼成可執行命令
- 逾時需可控並可中止

---

## 6. 最低合規檢查清單

每次改動前後至少確認：

- [ ] 程式碼與文件中無硬寫 token/password
- [ ] 日誌與錯誤訊息無敏感資訊明文
- [ ] 認證資訊未落盤
- [ ] API 錯誤回應符合統一格式
- [ ] 外部命令呼叫方式未引入注入風險

---
